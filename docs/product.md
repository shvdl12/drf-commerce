# 상품 서비스 설계 문서 (Product Service)


## 1. 개요

상품 서비스는 상품 정보 관리와 **재고의 정확한 동시성 제어**를 핵심 도메인으로 담당합니다. 특히 대규모 프로모션이나 타임 세일 상황에서도 시스템의 가용성을 유지하고 데이터 정합성을 보장하기
위해 **Redis 선점 후 DB 후행 반영**이라는 하이브리드 아키텍처를 채택하고 있습니다.


## 2. 핵심 도메인 모델

### 2-1. 상품


- **Status**: ON_SALE(판매중), OUT_OF_STOCK(품절), STOPPED(판매중지).
- **Price**: 정가와 할인가를 구분하여 관리.

### 2-2. 상품 재고


- **MySQL (Single Source of Truth)**: 최종 결제가 완료된 확정 재고를 관리합니다.
- **Redis (Performance Layer)**: 주문 생성 단계에서 실시간 경합을 처리하는 선점 재고를 관리합니다.

## 3. 재고 동시성 제어 전략


대량의 주문이 동시에 몰리는 상황에서 DB의 Lock 점유 시간을 최소화하고 응답 속도를 높이기 위해 재고 처리를 두 단계(Two-Phase)로 분리했습니다.


### 3-1. 1단계: 재고 선점 (Reservation - Redis)


- **시점**: 주문서 작성 및 결제 진입 시 (`OrderFacade.createOrder`)
- **기술**: Redis Lua Script
- **설계 결정**:
  - 다수 상품의 주문 건에 대해 원자성을 보장하기 위해 리스트 형태로 요청을 처리합니다.
  - Lua 스크립트에 주문번호(OID), 상품 ID(PID)와 수량(QTY)을 가변인자로 전달합니다.
  - 별도의 롤백 로직이 필요 없도록 **'전체 수량 체크 후 차감'** 전략을 사용합니다.
    1. **Pre-Check**: 모든 상품의 가용 재고(`product:stock:{pid}`)가 요청 수량 이상인지 먼저 전수 검사합니다. 하나라도 부족할 경우 즉시 실패를 반환합니다.
    2. **Atomic Execution**: 모든 상품이 유효할 경우에만 루프를 돌며 아래 작업을 일괄 수행합니다.
      - `DECRBY product:stock:{pid} {qty}`: 실시간 가용 재고 차감.
      - `HSET order:resv:{oid} {pid} {qty}`: 주문 중심 증빙 저장 (보상 스케줄러용).
      - `HSET product:resv_list:{pid} {oid} {qty}`: 상품 중심 인덱스 저장 (새벽 보정 배치용).

### 3-2. 2단계: 재고 확정 (Confirmation - MySQL)


- **시점**: 결제 완료 이벤트 소비 시 (`OrderEventProcessor.processPaymentCompleted`)
- **기술**: DB 원자적 업데이트 및 Redis 증명 제거
- **설계 결정**:
  - DB 업데이트 시 `WHERE stock >= :quantity` 조건을 포함하여 최종 정합성 방어.
  - DB 재고 차감 트랜잭션 **내부**에서 레디스 증빙 삭제를 수행합니다.
    1. DB 트랜잭션 시작 및 재고 차감 (`UPDATE ...`)
    2. 레디스 증빙 삭제 (`HDEL product:resv_list`, `DEL order:resv`)
    3. DB 트랜잭션 커밋

    - **사유**: 보정 배치가 `SELECT ... FOR UPDATE`로 락을 걸었을 때, 재고가 레디스에서 DB로 넘어가는 흐름을 완벽히 동결(Freeze)하기 위함입니다.

## 4. 재고 흐름 및 보상 트랜잭션


주문 프로세스 장애 시 **레디스 내 증빙 데이터**를 기반으로 자력 복구(Self-healing)를 수행합니다.

| 상황            | 대응 로직                            | 처리 방식                    |
|:--------------|:---------------------------------|:-------------------------|
| **재고 부족**     | 주문 생성 즉시 차단                      | 동기 (Exception)           |
| **쿠폰/결제 실패**  | 선점된 Redis 재고 복구 및 증빙 삭제          | 동기 (OrderFacade 내 catch) |
| **주문 취소/반품**  | Redis 및 DB 재고 모두 복구              | 비동기 (Kafka Event)        |
| **결제 미완료 만료** | **order:resv** 기반 증빙 존재 시에만 복구     | 배치 (Scheduler)           |

## 5. 상세 설계 포인트


### 5-1. 재고 동시성 제어 전략 비교 및 채택 이유


재고 관리의 정합성과 성능을 모두 잡기 위해 검토했던 3가지 전략을 비교합니다.


| 분류         | 1) Atomic SQL Update                | 2) Distributed Lock + DB             | 3) Redis Lua Reservation (채택)             |
|:-----------|:------------------------------------|:-------------------------------------|:------------------------------------------|
| **관리 주체**  | **MySQL (Single)**                  | **MySQL (Single)**                   | **Redis + MySQL (Hybrid)**                |
| **동시성 제어** | DB Row Lock (비관적)                   | Redis Distributed Lock               | Redis Lua Script (Atomic)                 |
| **장점**     | 구현이 단순하고 정합성이 완벽함.                  | 복잡한 로직 보호 및 DB 락 경합 전 줄 세우기 가능.      | **최고 수준의 처리량(Throughput)**. DB 트래픽 원천 차단. |
| **단점**     | **DB 커넥션 고갈** 및 행 잠금 경합으로 인한 성능 저하. | 락 획득/해제 오버헤드. 여전히 모든 요청이 DB I/O를 유발. | 두 저장소 간 정합성 관리(보상 트랜잭션) 비용 발생.            |

## 6. 리스크 관리 및 고도화 계획

### 6-1. 주문 프로세스 장애 및 재고 정합성

- **리스크**: 주문서만 생성되고 결제 전 장애로 고립된 `PENDING` 주문에 대한 재고 유실 또는 중복 복구.
- **대응 전략**
  - 주문 서버의 상태가 아닌, 상품 서버가 가진 `order:resv:{oid}` 증빙을 기준으로 복구 여부를 결정합니다.
  - 보상 시 해당 증빙이 존재할 때만 재고를 복구(`INCRBY`)하고 증빙을 삭제함으로써, 선점되지 않은 건에 대한 오복구를 원천 차단합니다.
  - 스케줄러를 통해 생성 후 일정 시간이 경과한 `PENDING` 주문을 탐색하여, 레디스 증빙 존재 시 안전하게 재고를 회수합니다.

### 6-2. 분산 환경의 이벤트 정합성

- **리스크**: 결제 완료 후 발행되는 Kafka 이벤트가 유실되어 DB 확정 재고가 차감되지 않거나, 네트워크 재시도로 인해 중복 소비되어 이중 차감되는 현상.
- **대응 전략**:
  - **발행**: **Transactional Outbox 패턴**을 적용하여 도메인 로직과 이벤트 저장을 하나의 로컬 트랜잭션으로 묶음으로써 이벤트 발행의 원자성을 확보하고
    Kafka로의 최소 1번 전송을 보장.
  - **소비**: **ProcessedEvent 멱등성 테이블**을 활용하여 이미 처리된 이벤트 ID를 체크하고, 중복 수신 시 로직을 스킵함으로써 Exactly-once 처리에
    준하는 데이터 정합성 유지.

### 6-3. 레디스 장애 시의 시스템 보호 전략

- **리스크**: 재고 선점 레이어인 Redis 장애 시, 모든 재고 요청이 DB로 전이되어 DB Row Lock 경합 및 커넥션 고갈로 인한 시스템 전체 연쇄 장애 유발.
- **대응 전략**:
  - **DB 보호 우선**: 대규모 트래픽 상황에서 Redis 장애 시 DB 직접 Failover는 시스템 전체를 마비시킬 수 있는 'Poison Pill'이 될 수 있으므로, 즉시 에러를 반환(
    Fail-fast)하여 DB 가용성을 최우선으로 보호.

### 6-4. 관리자 재고 보정 전략

- **리스크**: 관리자가 실사 재고를 반영하는 도중 실시간 주문 발생으로 인한 수치 불일치.
- **대응 전략**:
  - 관리자에게 '최종 목표 수량 설정(Absolute)'과 '단순 증감분 추가' 기능 모두 제공.
  - 수정 시 `SELECT ... FOR UPDATE`를 사용하여 DB 행을 점유.
  - 내부적으로는 항상 `(목표 수량 - 현재 DB 수량)`인 Delta 값을 산출하여 DB와 Redis에 각각 증감분 방식(`INCRBY`)으로 반영.

### 6-5. 정합성 보정 자동화

- **목적**: 누적된 미세 오차를 제거하고 DB(진실의 원천)와 Redis의 상태를 강제 동기화.
- **실행 전략**:
    1. 특정 상품에 대해 `SELECT ... FOR UPDATE`로 DB 행 잠금 획득. 이로써 해당 상품의 '결제 완료(DB 반영)' 흐름을 일시 동결합니다.
    2. 고정된 DB 재고($S_{db}$)와 Redis 선점 리스트($Q_{proof}$)를 조회하여 `기대값 = S_{db} - Q_{proof}` 산출.
    3. Lua 스크립트를 통해 `(기대값 - 실시간 Redis 재고)` 만큼만 `INCRBY` 반영.
    4. 트랜잭션 종료 및 락 해제.
- **설계 장점**:
  - DB 락을 통해 재고가 Redis에서 DB로 전이되는 찰나의 순간을 고정함으로써, 분산 환경에서도 완벽하게 일치하는 스냅샷을 기반으로 보정 수행.
  - 실시간 '주문 선점' 트래픽은 DB 락과 무관하게 Redis에서 중단 없이 처리됨.

### 6-6. 배치 처리 시의 부분 실패

- **리스크**: 다량의 상품을 동시 처리(Batch Reserve/Release)할 때, 일부 항목의 재고 부족이나 기술적 오류로 데이터 정합성이 깨지는 현상.
- **대응 전략**:
  - **선점**:
    - Lua 스크립트를 활용하여 모든 상품의 재고가 충분할 때만 일괄 차감. 하나라도 부족하면 전체를 실패 처리하여 데이터 파편화 방지.
  - **보상**:
    - 보상 트랜잭션은 '증빙 기반 멱등성'을 활용하여 개별 항목 단위로 성공할 때까지 반복 시도.
    - 특정 항목이 이미 복구되었거나 선점된 적 없는 경우를 '성공'으로 간주하고 조용히 다음 항목을 처리(Silent Success).
  - **애플리케이션 레벨 Fail-fast**: Redis 통신 장애 등 기술적 오류 발생 시 즉시 상위 레이어에 예외를 전파하여 보상 배치가 재시도할 수 있는 근거 마련.


## 7. Redis 키 설계 규칙

- **Stock Key (String)**: `product:stock:{productId}` (Value: Integer)
- **Order-Centric Proof (Hash)**: `order:resv:{orderId}`
  - **Field**: `{productId}` / **Value**: `{quantity}`
- **Product-Centric Index (Hash)**: `product:resv_list:{productId}`
  - **Field**: `{orderId}` / **Value**: `{quantity}`
- **Idempotency Key (String)**: `product:idempotency:{scope}:{key}` (API 중복 방지용)
